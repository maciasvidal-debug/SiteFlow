## 2025-03-05 - Fix security function roles configuration
**Learning:** Functions created without a definitive search_path parameter are susceptible to 'Role Mutable Search Path' warnings in PostgreSQL. If the caller alters the search path, they can inadvertently or maliciously invoke functions on unrelated schemas or alter operation outcomes.
**Action:** Recreated `get_my_role` and `get_my_department` functions adding `SET search_path = public` to enforce predictable schema resolution and comply with Supabase's Security linter rules.

## 2025-01-20 - Unsanitized Output in HTML InnerHTML Injection
**Vulnerability:** Data injected directly into `innerHTML` strings without proper sanitization. In `app.js`, various variables like `entry.hours`, `entry.minutes`, and even `entry.id` are concatenated directly into strings representing HTML, which are then assigned to `innerHTML`. This exposes the application to Cross-Site Scripting (XSS) attacks if an attacker can manipulate these supposedly 'safe' fields (like numeric IDs or durations) via direct API calls to Supabase or other means. Furthermore, using inline JavaScript handlers like `onclick` with HTML-escaped arguments reintroduces the vulnerability since HTML entities are decoded *before* the script executes.
**Learning:** Even fields that are expected to be numeric or system-generated IDs must be sanitized before being rendered into the DOM using `innerHTML` to follow Defense-in-Depth principles. Client-side validation can always be bypassed.
**Prevention:** Always use the provided `escapeHTML` utility function (or equivalent sanitization libraries) on ALL dynamic data before interpolating it into HTML strings, regardless of the expected data type. For interactive elements, NEVER use inline `onclick` handlers with dynamic arguments; instead, attach the data using `data-*` attributes and register event listeners that retrieve the value safely. Consider using `textContent` instead of `innerHTML` where HTML rendering is not strictly necessary.
